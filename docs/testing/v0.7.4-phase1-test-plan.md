# v0.7.4 Phase 1 UI Foundation Test Plan

Phase 1 adds shared frontend building blocks only. It must not change dashboard behavior, task editor behavior, import wizard behavior, storage, imports, exports, NFC handling, or backend APIs.

## Scope

- Shared status color tokens and status chip styles.
- Shared task status chip renderer.
- Compact task row renderer.
- Reusable section header renderer.
- Reusable dialog layout renderer.
- Dashboard metric card renderer.

## Manual checks

1. Open the HMM panel and confirm the dashboard loads without console errors.
2. Confirm the dashboard still shows the same four top metrics:
   - Maintenance health
   - Needs attention
   - Coming soon
   - On track
3. Confirm dashboard metric counts match task data before and after refresh.
4. Confirm dashboard category cards and Next up task cards still render.
5. Open an existing task from the dashboard and confirm the task detail dialog opens and closes normally.
6. Edit an existing task and confirm the task editor fields, conditional schedule sections, and save/close behavior are unchanged.
7. Open Settings import/export controls and confirm the import wizard still opens, navigates, and preserves selected review state.
8. Check the dashboard and task list at desktop width and mobile width. Verify chips, cards, progress bars, and buttons do not overlap.

## Status visual checks

Use sample tasks or existing task states to verify:

- OK/healthy status uses green treatment.
- Upcoming/due soon status uses amber treatment.
- Due status uses orange treatment.
- Overdue status uses red treatment.
- Paused, snoozed, and season paused statuses use muted treatment.
- Status text or icon remains visible alongside color.

## Regression checks

Run:

```bash
python -m compileall custom_components/home_maintenance_manager
```

Expected result: no Python compile errors.

If available in the local environment, also run the existing test suite:

```bash
pytest
```

Expected result: existing tests pass. Any failure should be investigated as a possible unrelated environment issue only after confirming the frontend file has no syntax errors in the browser console.
