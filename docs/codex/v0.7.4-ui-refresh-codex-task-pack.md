## Overall implementation goals

Implement a v0.7.4 UI refresh foundation that:

1. Makes the dashboard show both Home Health and Things Requiring Attention.
2. Adds or improves a compact task view for high task counts.
3. Refreshes task cards/rows around visual indicators and progress bars.
4. Updates the task editor language from Maintenance Trigger to Maintenance Rule.
5. Places Seasonal Restrictions directly below Maintenance Rules.
6. Reworks import entity configuration into a task-by-task Configure Tasks workflow.
7. Uses native Home Assistant entity pickers filtered by relevant entity domains.
8. Removes grouped/shared entity mapping UI and suggested entity lists from the import flow.

## Non-goals / do not implement now

- Do not add AI recommendations.
- Do not add weather scheduling.
- Do not add a full analytics engine.
- Do not add table view.
- Do not perform major backend storage migration unless required to preserve existing functionality.
- Do not remove existing task pack schema compatibility.
- Do not break import/export regression behavior.
- Do not break NFC behavior.

- Existing seasonal config already appears in `task.seasonal`.

Prefer reusing current state fields where practical, but rename UI concepts to Maintenance Rule.

# Task 7: Import wizard redesign

## Instructions

Replace the grouped/shared entity mapping experience with a task-by-task Configure Tasks workflow.

New wizard steps:

1. Select Tasks
2. Configure Tasks
3. Review Import
4. Import

Important behavior:

- No grouped entity queue.
- No shared entity requirement mapping screen.
- No suggested entity lists.
- User walks through each selected task that needs entity configuration.
- Required and optional entities are shown inside that task’s context.
- Entity chooser uses native Home Assistant `ha-entity-picker`.
- Entity picker is filtered by relevant domains where requirement data supports it.
- Optional entities can be skipped.
- Required entities should block progress or explicitly mark the task as importing paused if existing backend behavior supports unresolved required entities.

## Suggested implementation approach

1. Build a derived list of selected tasks requiring configuration from the import preview.
2. For each task, include the task’s entity requirements from preview data.
3. Store mappings per task requirement, not as a shared grouped entity queue.
4. Convert per-task mappings back into the backend import payload format at final import.
5. Preserve existing backend compatibility where possible.
6. If backend currently requires grouped mapping payloads, keep that internal translation hidden from the UI.

## Acceptance criteria

- Import wizard has four steps.
- Step 2 is named Configure Tasks.
- Entity mapping is task-by-task.
- Each task configuration screen shows task name, category, description, and relevant entity fields.
- Native HA entity picker is used.
- Suggested entities are removed.
- Grouped queue UI is removed from user-facing workflow.
- Selected tasks remain selected across steps.
- Review screen accurately summarizes entity assignments.

## Test plan

- Import a pack with no entity requirements.
- Import a pack with one required entity.
- Import a pack with one optional entity.
- Import a pack with multiple selected tasks requiring different entities.
- Select the same HA entity for two different tasks and confirm the UI allows it task-by-task.
- Skip optional entities.
- Try to continue without a required entity and verify expected behavior.
- Complete import and verify tasks were created with correct entities.
